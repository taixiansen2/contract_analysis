# 2026-04 · Week 3· 实验卡

> **研究问题**：Solana 的微观结构特性如何支撑 Prop AMM（以 HumidiFi 为例）实现优于 CEX / 传统 AMM 的报价？
>
> **核心思路**：以 HumidiFi 的"改价交易"（oracle update）为切入点，量化改价行为与报价质量的关系，并归因到 Solana 的链级特性。

---

## 实验 0：数据准备

**目标**：构建统一分析数据集，为后续所有实验提供基础。

**输入**


| 数据                        | 来源                                      | 说明                                     |
| ------------------------- | --------------------------------------- | -------------------------------------- |
| HumidiFi oracle update 交易 | Solana 链上（Helius RPC / Flipside / Dune） | 筛选 Program Address `9H6tu…3q6Rp` 的改价交易 |
| HumidiFi swap 成交交易        | 同上                                      | 同一 Program 下的用户成交记录                    |
| 传统 AMM 成交交易               | Orca / Raydium 的 SOL/USDC 池             | 作为链内对照组                                |
| CEX 基准价格                  | Binance API（SOL/USDC 逐秒 K 线）            | 作为"真实价格"基准                             |
| 时间窗口                      | —                                       | 选取近期连续 7 天，含高波动和低波动日                   |


**步骤**

1. 用 Dune 或 Flipside 写 SQL，拉取 HumidiFi 在目标时间窗口内的所有 oracle update 交易，字段包括：`slot`、`block_time`、`tx_hash`、`CU_consumed`、`tip_amount`、`成功/revert`、更新后的价格参数。
2. 同样拉取 HumidiFi 的所有 swap 交易，字段包括：`slot`、`block_time`、`tx_hash`、`成交价`、`成交量`、`交易方向（买/卖）`。
3. 拉取 Orca/Raydium SOL/USDC 池在同一时间窗口的 swap 交易，字段同上。
4. 从 Binance API 拉取对应时间段 SOL/USDC 的逐秒收盘价。
5. 将所有数据按 `block_time` 对齐到秒级时间戳，合并为一张宽表。每笔 swap 匹配：(a) 最近一次 oracle update 的时间和价格，(b) 同一秒的 Binance 中间价。
6. 计算基础指标：`报价偏差 = |成交价 - Binance 基准价| / Binance 基准价`。

**输出**

- 一张清洗好的分析表（CSV），每行一笔 swap，列包括：时间、成交价、CEX 基准价、报价偏差、成交量、交易方向、AMM 类型（HumidiFi / Orca / Raydium）、距上次 oracle update 的时间差。
- 一份数据质量检查报告：样本量、缺失值、时间对齐精度。

## 实验 1：改价频率与报价质量

**假设**：HumidiFi 改价频率越高，其报价偏差越小（即报价越贴近 CEX 真实价格）。

**输入**

- 实验 0 的分析表
- HumidiFi oracle update 记录

**步骤**

1. 对每笔 HumidiFi swap，计算 `距上次改价的时间差 Δt`（单位：毫秒或 slot 数）。
2. 将所有 swap 按 Δt 分桶（如 0-400ms / 400-800ms / 800-1200ms / >1200ms）。
3. 计算每个桶内的平均报价偏差和中位数报价偏差。
4. 画散点图：X 轴 = Δt，Y 轴 = 报价偏差。叠加分桶均值的阶梯线。
5. 跑简单 OLS 回归：`报价偏差 ~ Δt`，看系数是否显著为正。
6. 加入对照组：对 Orca/Raydium 的 swap 做同样分析（传统 AMM 没有主动改价，可用"距上次被交易的时间"替代）。

**输出**

- 散点图 + 分桶均值图，展示 Δt 与报价偏差的关系。
- 回归结果表（系数、p 值、R²）。
- 一句话结论：改价频率对报价质量的边际贡献有多大。

## 实验 2：改价成本与 Solana CU 效率

**假设**：HumidiFi 的极低 CU 消耗（~50-143 CU）使其能以极低成本实现高频改价，这是 Solana 微观结构的直接支撑。

**输入**

- HumidiFi oracle update 交易（含 CU 消耗、tip 金额）
- 其他 Prop AMM（SolFi、TesseraV）的 oracle update 交易（如可获取）
- Solana 基础费用参数

**步骤**

1. 统计 HumidiFi 每笔 oracle update 的 CU 消耗分布（直方图）。
2. 计算每笔 oracle update 的实际成本（lamports），换算成美元。
3. 计算每日的总改价成本和改价次数，得到"单次改价成本"和"每日改价预算"。
4. 如果能获取 SolFi（~~4339 CU）和 TesseraV（~~1595 CU）的数据，做横向对比：同样的改价频率下，三者的日成本差异。
5. 做反事实估算：如果 HumidiFi 的改价逻辑放在以太坊上（假设每次改价消耗 ~100,000 gas，gas price 取近期均值），同等频率下日成本是多少？
6. 计算 Jito 拍卖中的 Tip-per-CU 比值，验证 HumidiFi 是否以低绝对成本获得了高优先级。

**输出**

- CU 消耗分布图。
- 成本对比表：HumidiFi vs 其他 Prop AMM vs 以太坊反事实。
- 一句话结论：Solana 的低 CU 成本为 HumidiFi 节省了多少改价成本，这种成本优势在以太坊上是否可复制。

## 实验 3：改价响应速度与价格跟踪

**假设**：当 CEX 价格发生突变时，HumidiFi 的 oracle update 能在 1-2 个 slot（400-800ms）内跟上，而传统 AMM 需要等待套利者交易才能更新价格。

**输入**

- Binance SOL/USDC 逐秒价格
- HumidiFi oracle update 时间序列
- Orca/Raydium 成交价时间序列

**步骤**

1. 识别 CEX 价格突变事件：找出 Binance 价格在 5 秒内变动超过 0.3% 的时刻（阈值可调）。
2. 对每个突变事件，记录：(a) 突变发生的时间 T₀，(b) HumidiFi 第一次 oracle update 落地的时间 T₁，(c) HumidiFi 报价偏差回到正常水平（如 <5bps）的时间 T₂。
3. 计算响应延迟 = T₁ - T₀，收敛时间 = T₂ - T₀。
4. 对 Orca/Raydium 做同样分析：找到突变后第一笔套利交易的时间，以及价格收敛时间。
5. 对比两组的延迟分布（箱线图）。
6. 将响应延迟换算为 slot 数，看是否接近 1-2 个 slot 的理论下限。

**输出**

- 价格突变事件案例图：X 轴=时间，Y 轴=价格，叠加 CEX 线、HumidiFi 报价线、传统 AMM 报价线。
- 响应延迟的箱线图对比。
- 一句话结论：HumidiFi 的价格跟踪速度比传统 AMM 快多少，是否接近 Solana 出块速度的物理上限。

## 实验 4：量价关系与动态定价

**假设**：HumidiFi 对不同成交量给出不同的报价（动态滑点），且其滑点结构优于传统 AMM 的固定曲线。

**输入**

- 实验 0 的分析表（含成交量字段）

**步骤**

1. 将所有 swap 按成交量分档：小单（<100 USD）、中单（100-1000 USD）、大单（1000-10000 USD）、超大单（>10000 USD），阈值根据数据分布调整。
2. 对每档计算：平均报价偏差、中位数报价偏差、平均滑点（成交价 vs 报价的偏差）。
3. 画分档柱状图：X 轴=成交量档位，Y 轴=报价偏差，按 AMM 类型分组。
4. 对 HumidiFi 和传统 AMM 分别拟合 `报价偏差 ~ log(成交量)` 的曲线。
5. 对比两条曲线的斜率：传统 AMM 的斜率应接近理论恒定乘积曲线的预测值，HumidiFi 如果有动态定价，斜率应更平缓（大单滑点更小）。

**输出**

- 分档柱状图。
- 量价关系拟合曲线对比图。
- 一句话结论：HumidiFi 的动态定价在哪个量级上优势最明显。

## 实验 5：改价失败（Revert）与逆向选择

**假设**：当 oracle update 失败（revert）时，HumidiFi 的报价变得陈旧，此时发生的成交更可能是有毒订单流（套利者），报价质量显著下降。

**输入**

- HumidiFi oracle update 记录（含成功/revert 状态）
- HumidiFi swap 成交记录
- CEX 基准价格

**步骤**

1. 标记每笔 swap 之前最近一次 oracle update 的状态：成功 or revert。
2. 将 swap 分为两组：A 组 = 前序改价成功，B 组 = 前序改价失败（或连续多次失败）。
3. 计算两组的平均报价偏差，做 t 检验看差异是否显著。
4. 对每笔 swap 计算"事后价格变动"：成交后 5 秒 / 30 秒 / 60 秒的 CEX 基准价相对成交价的变动方向。如果成交后价格朝对交易者有利的方向动了，标记为"有毒交易"（informed trade）。
5. 比较 A 组和 B 组中有毒交易的占比。
6. 估算 revert 导致的额外损失：B 组的平均报价偏差 × B 组成交量总和。

**输出**

- A/B 两组报价偏差的箱线图对比。
- 有毒交易占比对比表。
- 估算的 revert 损失金额。
- 一句话结论：改价失败对报价质量的影响有多大，验证"Solana 上改价可靠性"的重要性。
