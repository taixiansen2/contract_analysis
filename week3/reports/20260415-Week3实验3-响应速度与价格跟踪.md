# Week 3 · 实验 3：改价响应速度与价格跟踪

- 时间窗口：2026-04-15 00:00:00 – 01:00:00 UTC（1 小时试点）
- 数据：`binance_sol_usdc_20260415.csv`（逐秒 SOL/USDC）、`humidifi_oracle_updates_20260415.csv`（31,461 笔 oracle update / 21,642 success）、`humidifi_swaps_20260415.csv`（4,998 笔非 dust 且 quote=USDC 的 swap）、`orca_raydium_sol_usdc_20260415_h0-1.csv`（Whirlpool 3,606 + Raydium 8,339 笔 SOL/USDC swap）

## 研究问题与假设

当 CEX（Binance）SOL/USDC 价格发生突变时，HumidiFi 的 oracle update 能否在 1-2 个 slot（400-800ms）内跟上？传统 AMM（Whirlpool / Raydium）需要多久才能通过套利交易把池内价格拉回 CEX 水准？

假设：HumidiFi 的响应延迟显著短于传统 AMM，且接近 Solana 出块速度的物理下限（1-2 slot）。

## 口径与定义

- **突变事件**：Binance close 价在 5s 滑窗内 `max/min - 1 ≥ threshold`，同一事件冷却 5s。
- **主阈值 0.10%/5s（10 bps）**，同时做 0.05%（24 事件）与 0.15%（5 事件）做敏感性。2026-04-15 是低波动日（24h max 5s 波动 = 0.32%），若沿用实验卡原始阈值 0.3%/5s，1h 内 0 个事件。
- **HumidiFi 响应 T1**：
  - `T1_oracle`：`t ≥ T0` 的第一笔成功 oracle update 的 `block_time`
  - `T1_swap`：第一笔 HumidiFi swap 的 `block_time`（作为"报价已更新并被使用"的侧证）
- **控制组响应 T1**：突变后第一笔 SOL/USDC swap 的时间（"套利者第一笔交易"的代理）
- **收敛 T2**：从 T0 开始，任意 AMM 第一笔 swap 其 `|price_usdc_per_sol / Binance.close(block_time) - 1| ≤ 5 bps`
- **HumidiFi 报价代理**：swap 成交价（`price_usdc_per_sol`），未解码 oracle update 的 `p0..p7`
- **时间分辨率**：Binance 1s + Solana `blockTime` 1s → 响应延迟分辨率 ±1s（≈2.5 slot）。"Slot 数"维度为 `delay_s / 0.4` 近似换算，仅作参考展示。

| Raydium   | response_swap   | 8   | 0.0 s     | 0.25 | 1.0 | 1.0 | 0.00          |
| Raydium   | convergence     | 8   | 0.5 s     | 0.63 | 1.3 | 2.0 | 1.25          |


要点：

- **HumidiFi oracle update 的响应延迟 median = 0 s、max = 0 s**（8 个事件无一例外都在 T0 所在的那一秒内有成功 oracle 落地）。受 1s 时间分辨率所限我们无法直接区分"同秒内 0 slot vs 1 slot"，但这已经触到 Binance K 线 + Solana `blockTime` 的共同下界。
- HumidiFi oracle update 在 1h 内共 31,461 笔（平均 **~115 ms / 笔**），因此在任意秒内至少命中一次 oracle 的概率接近 100%。这正是"**报价无条件重推**"模式的直接观测证据。
- 控制组 Whirlpool 首笔 swap 延迟 median = 0.5s、max = 4s，Raydium 因流量更大 median = 0s。这里"响应"指的是"突变后第一笔 SOL/USDC 成交时间"，与 HumidiFi 的"主动改价"在机制上**不等价**——HumidiFi 不依赖任何外部成交就能改价，而传统 AMM 必须等套利者来交易。
- 收敛时间上，HumidiFi 与 Whirlpool 都是 1.5s median；Raydium 因流量最大（2.3 swap/s），每秒都能采样到接近 CEX 的成交价，看起来也非常快。这里收敛判定主要反映了**该 AMM 的成交频率**，并不能直接证明该 AMM"跟得更快"——因此更有说服力的指标是 HumidiFi 的 `response_oracle`。

## 一句话结论

在 2026-04-15 00:00-01:00 UTC 这一 1h 低波动试点窗口中，HumidiFi 针对 ≥10 bps 的 CEX 突变事件，其 oracle update 的响应延迟中位数 = **0 s**（≈ 0 slot，受限于 1 秒时间分辨率），表现出的是"报价无条件重推、不等待套利者"的微观结构特征，与实验卡提出的"1-2 slot 内跟上"的假设一致；传统 AMM 的首笔套利交易虽然受其自身池流量影响能在同秒内出现，但其改价本质上仍依赖外部成交触发。

## 已知限制

- **1s 时间分辨率**：Binance K 线与 Solana `blockTime` 都是 1s，无法直接区分"0 slot vs 1 slot"。要进到亚秒级需要逐笔 Binance trade stream + oracle 的 slot 号对齐，本轮超出范围。
- **低波动日 + 阈值下调**：为了在 1h 内产生足够事件，突变阈值从 0.3%/5s 下调到 0.1%/5s；事件相对温和，结论对大幅突变（例如新闻冲击下的 1% 以上跳变）的外推需要谨慎。
- **HumidiFi "报价线"代理**：采用 swap 成交价而非解码 oracle 参数。swap 采样稀疏会轻度拉长 T2 测量。
- **控制组"响应"非对等**：Whirlpool/Raydium 没有主动改价，我们用"首笔 swap 时间"代理，受该 AMM 当时成交密度影响较大；Raydium 因流量最大（2.3 swap/s）被"拉得很快"，但这并不等价于它的定价对 CEX 敏感。
- **Orca V1 缺席**：Dune `project='orca'` 在 1h 内无 SOL/USDC 成交，只保留 Whirlpool / Raydium。
- **单日 + 单小时**：未跨窗口/多日验证。若要给出置信区间与分布尾部画像，需扩到整天或多日并引入抓取自动化。
